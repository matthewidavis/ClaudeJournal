## 1. ClaudeJournal title should always navigate Home
Clicking the **"ClaudeJournal"** site-header title currently doesn't go
anywhere. Expected: it acts as the canonical Home link from any page —
deep-link, filtered, or pristine. Anchor the `<h1>` text to
`anchor_base + 'index.html'` (with the same in-place reset behavior on
the home page that the Clear chip uses, or just leave it as a plain
navigation since the title-click is unambiguous).

## 2. Clear chip no longer at the edge
After Phase A added the **Graph** chip and Phase B added **Loops** and
**Learnings** chips to the modes row, the Clear chip is no longer the
last child — so it doesn't read as the "end of the row" affordance any
more. Either (a) keep Clear as the *trailing* element regardless of
what gets added later (push it to last position in render), or (b)
move Clear to its own visual zone (right-aligned with `margin-left:
auto` or a divider) so it doesn't get jostled by new chips.

## 3. Modes-row reorganization needed
With Graph, Loops, Learnings, Echoes layered onto the original Find /
Daily / Weekly / Monthly / Library set, Row 0 is getting busy. Worth a
deliberate reorg — possibly split into two visual groups: navigation
(Graph, Loops, Learnings, Echoes — these go to standalone pages) vs.
filter-state (Find + view chips). Clear stays trailing in either case.

## 4. "Aha moment" axis appears non-functional
Selecting **Yes / No** under the Aha-moment axis appears to surface the
same documents regardless of which option is clicked. Verify the
`data-learning="yes"|"no"` attribute is being set correctly on
day entries (templates.py:2843ish — the ternary `"yes" if has_learning
else "no"`) AND that the `has_learning` source signal isn't accidentally
true for every brief. Likely a bug in how `has_learning` is computed
upstream from the brief's `learned` field — it might be
`bool(learned_list) is always True` because the field always exists as
an empty list rather than None.

## 5. Graph view feels like a separate page, not a journal feature
The graph view (`out/graph.html`) lives as its own page — opening it
swaps the whole document. Two threads worth pulling:
  - **Embedded experience:** consider a graph *modal* or a graph view
    that renders inside the main feed page (toggleable like Daily ↔
    Library), so it feels like a lens on the journal rather than a
    separate destination.
  - **Filter-aware graph:** if the user has filtered the feed (e.g.,
    Project=ChromaKey + Topic=architecture), clicking Graph should
    open the graph **with the same filter applied** — so the graph
    view shows the subset they're looking at, not the full corpus.
    Currently Graph is a hard navigation that drops the filter state.

## 6. Add search to Learnings (like briefs/prompts/moments)
Daily-entry inspect chips (`briefs`, `prompts`, `moments`, `files`)
each have an inline search box that filters their content. The
**Learnings** standalone page has no equivalent. Add the same
`<input class="inspect-search" type="search" placeholder="filter
learnings...">` pattern so the page is navigable with a corpus this
size (currently 291 entries; will grow).

## 7. Loops chip doesn't get an "active" state, no Clear shown
When a user clicks **Loops** (or Learnings, Graph, Echoes — anything
that navigates to a standalone page), three problems:
  - The chip doesn't render with `.active` styling on the destination
    page, so the user has no visual confirmation of where they are.
  - The Clear chip is hidden because the destination page is "deep-link"
    with no filters active in the modes row — but the user IS in a
    filtered view (just navigated, not state-filtered). They have no
    way back except the browser back button.
  - On Loops/Learnings/Echoes pages, the modes-row chips for the
    *other* nav targets still show — Loops/Learnings/Graph/Echoes
    chips are present, but clicking them re-navigates rather than
    filtering. That's fine, but Clear should always be present on
    these pages with "go home" semantics, the same way it is on
    project-arc / topic / doc deep-link pages.

## 8. Entity inspect chip is confusing in current shape
The new **Entity** chip on daily entries (added in C5) shows entities
grouped by type (People / AI Models / Libraries / Services) but the
UX reads like a project-arc summary or daily summary, not a list of
extracted entity mentions. Two interpretations of what's wrong:
  - **Possibility A:** the chip is rendering content from the wrong
    source (maybe pulling brief.summary instead of brief_entities
    join). Verify the data pipeline: brief → entity extraction →
    brief_entities → render path.
  - **Possibility B:** the chip is rendering correctly but the visual
    presentation looks too narrative — needs a tighter chip-grid
    layout (small pills clustered by type) rather than what currently
    looks like flowing text.

User said: *"seems kind of like project arcs but not sometimes
components of projects but only displays one summary that seems a like
a daily? a little confused on this one"* — strongly suggests A
(rendering wrong content) over B (just visual polish).

## 9. `v5` prompt-bump comment is factually wrong
At `claudejournal/narrator/claude_code.py` lines 87–92 the comment
claims dates without annotations are "unaffected by the version bump
because the input hash includes the annotation content (empty list →
same hash as before)." This is wrong. `NARRATION_PROMPT_VERSION` is
folded into the hash unconditionally (`narrate.py` line 103), so v4
and v5 produce different digests for identical briefs even with zero
annotations. The next pipeline run regenerates **every** daily
narration, not just annotated ones. Fix: either correct the comment to
say so (cheap), or add a guard that only bumps the version string when
annotations are present for the date (cleaner, more invasive). For a
journal this size the cost is one sonnet sweep — annoying, not
catastrophic, but the misleading comment will confuse future readers.

## 10. Phase A commit message misattributes new code as pre-existing
The Phase A commit (`d59c132`) message says
`claudejournal/db.py: links table schema (already in place)`, but
`git show` confirms the `links` table DDL, both indexes, and the
entire `backlinks.py` module were all NEW in that commit. No code
impact — only `git log` cosmetics — but worth fixing if/when a related
follow-up commit lands (could amend in a related PR or just add a
clarifying note in the next phase's commit message).

## 11. `_rebuild_links` runs unconditionally on every render
The link graph is truncated and re-inserted from all narration prose
on every render call, with no freshness check. Plan accepted this as
"idempotent, cheap" and the table is small today, but as the corpus
grows this becomes O(n²) work on every render. Future optimization
target: gate `_rebuild_links()` behind a hash check over
`narrations.input_hash` values, or maintain links as an incremental
write during narration generation rather than a render-time rebuild.
